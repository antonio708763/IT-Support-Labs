# AGENTS.md

## Repository Purpose

This repository is the main IT support portfolio for Antonio's employer-facing lab work. It documents hands-on practice with VirtualBox Active Directory, a Proxmox homelab, Action1 endpoint management, and related help desk skills.

Any automated assistant, AI agent, or contributor working in this repository should help keep the portfolio accurate, professional, and easy for hiring managers or technical reviewers to understand.

## Writing Style Rules

- Use a clear, professional, first-person voice when describing Antonio's work.
- Write for employers, recruiters, and IT managers who may be reviewing the portfolio quickly.
- Keep explanations practical and specific: describe what was built, configured, tested, fixed, or learned.
- Prefer plain language over buzzwords.
- Use active voice where it improves clarity.
- Avoid filler, hype, or vague claims such as "expert-level," "enterprise-grade," or "production-ready" unless the documentation proves that level of work.
- Expand acronyms on first use when the audience may not know them.
- Keep formatting consistent across lab writeups, including headings, lists, screenshots, and command examples.

## Documentation Standards

Each lab or project document should make the scope easy to understand. When relevant, include:

- The goal of the lab or exercise.
- The tools, platforms, and versions used.
- The environment setup, including virtual machines, network ranges, roles, and services.
- Step-by-step configuration notes that another learner could follow.
- Screenshots or diagrams when they clarify the setup or result.
- Troubleshooting notes, including errors encountered and how they were resolved.
- A short skills summary connecting the lab to help desk, desktop support, system administration, or endpoint management work.
- Any known limitations, unfinished work, or future improvements.

Use Markdown headings in a predictable order. Keep commands and configuration snippets in fenced code blocks. Do not include large unrelated logs unless they directly support the lab result.

## Portfolio Accuracy Rules

- Do not exaggerate Antonio's experience, job titles, certifications, or production responsibilities.
- Do not imply paid professional experience where the work was completed in a personal lab.
- Do not claim that a lab environment is equivalent to managing a real enterprise environment.
- It is acceptable to describe transferable skills, such as user account management, Group Policy practice, endpoint patching concepts, ticket-style troubleshooting, virtualization, networking basics, and documentation.
- Be honest about the learning context. Phrases like "practiced," "configured in a lab," "documented," "tested," and "simulated" are preferred when they accurately describe the work.

## Repo Safety Rules

- Do not publish secrets, passwords, API keys, tokens, license keys, private URLs, private IP details that should remain confidential, or personal identifying information.
- Redact sensitive values in screenshots, logs, exports, and configuration examples before committing them.
- Do not commit real customer, employer, coworker, or end-user data.
- Do not include credentials for Active Directory, Proxmox, Action1, GitHub, cloud services, routers, firewalls, or any other system.
- Use placeholders such as `<REDACTED>`, `<LAB_DOMAIN>`, `<USERNAME>`, or `<PRIVATE_IP>` when examples need sensitive-looking values.
- Review screenshots carefully for visible account names, email addresses, device serial numbers, hostnames, tokens, browser tabs, and notification popups.
- Do not add files generated by tools unless they are useful documentation artifacts and safe to publish.

## Change Guidelines

- Keep changes focused on the requested lab, document, or portfolio improvement.
- Preserve the existing meaning of Antonio's work when editing for grammar or structure.
- Do not delete lab notes, screenshots, or project history unless the user explicitly asks.
- If information is missing, add a clear placeholder or note instead of inventing details.
- When creating new documentation, prefer a structure that can be reused across future labs.
- Check links, image paths, and Markdown rendering before considering documentation complete.

## Lab-Specific Notes

- VirtualBox Active Directory lab documentation should focus on domain setup, user and group management, Group Policy practice, DNS/DHCP concepts, Windows client joining, shared resources, and troubleshooting.
- Proxmox homelab documentation should focus on virtualization, VM/container organization, storage and networking choices, backup habits, resource planning, and lessons learned.
- Action1 endpoint management documentation should focus on endpoint inventory, patch management practice, remote management concepts, reporting, automation, and security hygiene.
- Help desk documentation should connect labs to practical support workflows such as intake, diagnosis, escalation notes, password/account tasks, endpoint maintenance, and clear user-facing communication.

## Pull Request Expectations

For documentation pull requests, include:

- A short summary of what changed.
- Why the change improves the portfolio.
- Any safety review performed, especially secret or screenshot checks.
- Any remaining placeholders or follow-up work.

Keep pull requests draft until Antonio has reviewed the wording and confirmed that the experience level is represented accurately.
